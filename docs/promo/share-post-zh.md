# dont-deal：一个给程序员准备的“别拿命继续 deal”技能

最近在做一个有点不一样的 skills 项目，叫 `dont-deal`。

它不是效率工具，不是代码补全，也不是又一个工作流插件。

它做的事情其实很简单：

当一个程序员在长期熬夜、连续高压工作的时候，如果突然胸口不舒服，模型能不能别再顺着“继续干活”这个方向说话，而是认真一点，保守一点，直接一点。

我想做的，就是这个。

## 这个项目是干什么的

`dont-deal` 是一个本地优先的 skills 项目，给 Codex、Claude Code、OpenClaw 这类 AI 编程环境用。

它主要做两件事：

1. 被动感知最近的工作状态

它会在用户授权后，只读当前仓库的 git 提交时间、当前系统时间和时区，粗略判断最近是不是在连续熬夜、短睡、硬扛。

不是为了监控，是为了给模型一点背景。

比如模型知道：

- 这个人最近三天都在凌晨还在提交
- 上一次提交是凌晨 3 点，下一次是早上 9 点
- 这个人可能根本没休息够

那它在面对胸痛、胸闷、后背疼这种描述时，就不该再轻描淡写。

2. 主动做一个非常克制的快速判断

当用户说自己胸口不对劲的时候，这个 skill 会先问几个关键问题，比如现在是不是还在痛、是闷是紧还是针扎、会不会往后背和下巴放、有没有出汗气短、已经持续多久了。

然后它再结合本地疲劳背景，去回答一个更实际的问题。

它不追求“诊断出到底是什么病”，它只解决一个问题：

- 现在是不是该立刻呼救
- 如果不是立刻呼救，是不是也必须今天去医院

只有两档：

- 红色：立刻呼救
- 黄色：今天尽快就医

没有“再观察观察”，没有“可能只是压力大”。

git 熬夜信息在这里不是单独判定依据，只是辅助模型在危险场景里更保守一点。

## 为什么要做这个

因为很多程序员真的会把身体报警，当成工作的一部分。

胸口闷、后背疼、冒冷汗、像牙疼一样往下巴放，很多人第一反应不是停下来，而是：

- 先把这段代码写完
- 先把这个 bug 修掉
- 先把这个活 deal 掉

这件事我越想越觉得不对。

很多时候，模型很懂技术，但并不知道用户最近到底是不是已经在透支。

所以我想做一个反过来的技能：

不是让 AI 帮你更猛地干活，
而是让 AI 在必要的时候提醒你，

> 这次别 deal 了，先保命。

## 这个名字是什么意思

`dont-deal` 不是单纯说“别处理事情”。

它真正想表达的是：

> 别为了把事情 deal 掉，继续拿自己的健康和命去硬换。

对程序员来说，“deal” 这个词太顺手了，什么都想 deal 一下。

但有些场景，真的不该再 deal。

## 它怎么用

项目已经开源在 GitHub：

[shaoqing404/dont-deal](https://github.com/shaoqing404/dont-deal)

本地跑起来很简单：

```bash
git clone git@github.com:shaoqing404/dont-deal.git
cd dont-deal
DONT_DEAL_HOME=./data npm run triage:quick
```

它会用中英双语开场，先问几个很短但关键的问题，再结合最近的疲劳背景，给出一个非常直接的结果。

另外，它也已经发到了 OpenClaw 的 Skills Hub，skill 名是：

`dont-deal-triage`

## 适合谁

- 经常熬夜写代码的人
- 长期高压工作的 AI 开发者
- 想给自己的编程环境加一点“健康保守机制”的人
- 想研究本地优先、AI + 健康提醒、技能化交互的人

## 最后

这个项目不是医疗器械，也不是医生。

但我觉得它至少可以做一件有意义的事：

当用户已经在危险边缘的时候，不要再帮他继续往前推。

如果你身边也有那种“胸口都开始不舒服了还想把活干完”的朋友，可以把这个项目发给他看看。

有些时候，真正应该被打断的，不是工作流，而是硬撑。
